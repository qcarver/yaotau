# yaotau

ESP-IDF OTA demo project with a reusable `yaotau` component.

## Project summary

This repository demonstrates a simple OTA update flow for ESP32 using dual OTA slots:

1. App starts in `main/app_main.c` and logs the generated build version.
2. `ota_helper_init()` runs at startup (non-fatal by design).
3. Helper initializes NVS/network/event loop, connects to Wi-Fi, and fetches remote update metadata.
4. Current build version is compared against server version using a basic semver comparator.
5. If remote version is newer, firmware is downloaded to the next OTA partition and boot partition is switched.
6. Device restarts into the new image.

If any OTA step fails, the helper logs the issue and returns without blocking the rest of the app.

## Key modules

- `components/yaotau/src/ota_helper.c`
  - Orchestrates full OTA check/update flow.
  - Logs app metadata and OTA partition info.
  - Calls Wi-Fi and HTTP helpers.
  - Reboots only after successful OTA write and boot partition switch.

- `components/yaotau/src/yaotau_wifi.c`
  - Connects in station mode using Kconfig-provided SSID/password.
  - Retries up to `CONFIG_YAOTAU_WIFI_MAX_RETRY`.
  - Uses an event group to wait for connect/fail outcome.

- `components/yaotau/src/yaotau_http.c`
  - Fetches `version.json` from `CONFIG_YAOTAU_SERVER_VERSION_URL`.
  - Parses JSON keys: `version`, `image_url`.
  - Streams image download and writes it via `esp_ota_write()`.
  - Validates HTTP status and image size against OTA partition size.

- `components/yaotau/src/yaotau_semver.c`
  - Compares versions as `x.y.z` using integer tuple comparison.

## Build/version pipeline

- `version.txt` stores the project version as `x.y.z`.
- On every build, `tools/yaota_bump_version.py` increments patch version and generates:
  - `build/generated/yaota_version.h` with `#define YAOTA_BUILD_VERSION "..."`
- Root `CMakeLists.txt` also creates OTA serving artifacts after build in `build/serve_me/`:
  - `the_app_image.bin` (copied from app output)
  - `version.json` generated by `tools/yaota_write_version_json.py`

## OTA server contract

Device expects a JSON document at the configured version URL:

```json
{
  "version": "1.2.3",
  "image_url": "http://<host>:<port>/the_app_image.bin"
}
```

`image_url` must point to a binary compatible with the target and partition layout.

## Partition layout

`partitions.csv` defines two OTA app slots (`ota_0`, `ota_1`) each sized `0x100000` (~1 MiB), plus `otadata` and standard NVS/PHY entries.

## Configuration

Use `idf.py menuconfig` and configure `yaotau (yet another over-the-air updater)`:

- `YAOTAU_ENABLE`
- Wi-Fi credentials (`YAOTAU_WIFI_SSID`, `YAOTAU_WIFI_PASSWORD`)
- Update metadata URL (`YAOTAU_SERVER_VERSION_URL`)
- HTTP timeout (`YAOTAU_HTTP_TIMEOUT_MS`)
- Image size guard (`YAOTAU_IMAGE_MAX_BYTES`)

Defaults are in `sdkconfig.defaults` and `components/yaotau/Kconfig`.

## Main app behavior

`main/app_main.c` is intentionally minimal:

- Logs startup/build version.
- Calls `ota_helper_init()` once.
- Continues a periodic heartbeat loop regardless of OTA outcome.

This makes the OTA helper safe to embed into larger applications without making boot success depend on update server availability.
